# Session Closer & Archivist

You are closing a development session. Your goal is to "pack up" the current session, ensuring all progress is documented, context is preserved, and the next session can start immediately without context-switching costs.

## Context Gathering

First, gather the current state:

### 1. Git State (Automatic)
**Current Branch:** !`git rev-parse --abbrev-ref HEAD 2>/dev/null || echo "Not a git repo"`

**Uncommitted Changes:**
!`git status --short 2>/dev/null || echo "N/A"`

**Staged Changes:**
!`git diff --cached --stat 2>/dev/null | tail -5 || echo "Nothing staged"`

**Unstaged Diff Summary:**
!`git diff --stat 2>/dev/null | tail -10 || echo "No unstaged changes"`

**Commits Made This Session (last 5):**
!`git log --oneline -5 --since="8 hours ago" 2>/dev/null || echo "No recent commits"`

### 2. Conversation Context
Review the current chat history for:
- Original session goals (what did we set out to do?)
- Problems encountered and solutions found
- Decisions made and their rationale
- Any "aha moments" or learnings
- Debugging steps that worked or didn't

### 3. Code State Detection
Look for indicators of incomplete work:
- Search for `TODO`, `FIXME`, `HACK`, `XXX` comments added today
- Check for commented-out code blocks
- Look for test files with failing or skipped tests
- Check for console.log/print statements left for debugging

## Analysis Workflow

1. **Summarize Achievements:** Compare what was done against original session goals
2. **Identify Mental State:** Capture non-code context (e.g., "Realized library X is too slow, pivoting to Y")
3. **Detect Loose Ends:** Identify broken tests, TODOs added, or half-finished logic
4. **Documentation Check:** Did scope change? Does PRD need updating?

## Output Format

Generate the handover report in this exact format:

```markdown
# Session Closure Report

**Date:** [Current date]
**Branch:** [branch name]
**Duration Context:** [If known from conversation]

## 1. Executive Summary

**Outcome:** [Succinct sentence: "Implemented Auth," "Debugged API," "Refactored data layer," etc.]

**Session Goals vs. Achieved:**
| Goal | Status |
|------|--------|
| [Original goal 1] | [Done/Partial/Blocked] |
| [Original goal 2] | [Done/Partial/Blocked] |

**Commit Recommendation:**
```
feat(scope): brief description

- Technical change 1
- Technical change 2
- Technical change 3
```

## 2. The "Cliffhanger" (CRITICAL)

> *What exactly is the state of the code right now?*

* **Build Status:** [Pick: Passing / Unstable / Broken / Unknown]
* **Test Status:** [All passing / X failing / Not run]
* **Uncommitted Work:** [Yes - X files / No - all committed]

**If Unstable/Broken:**
```
[Paste the specific error message or describe the logic flaw]
```

**Mental Context:** [Why did we stop here?]
- [e.g., "End of available time"]
- [e.g., "Blocked waiting for API credentials"]
- [e.g., "Need to research X before continuing"]
- [e.g., "Hit unexpected complexity, need fresh perspective"]

**Last Thing Attempted:**
[What was the last action/change/test before closing?]

## After Generating Report

1. **Offer to save:** Ask if user wants the report saved to a file (suggest `SESSION_LOG.md` or `docs/sessions/YYYY-MM-DD.md`)

2. **Commit reminder:** If there are uncommitted changes, remind user to commit with the suggested message

3. **Cleanup reminder:** If debugging artifacts were found (console.logs, etc.), ask if they should be removed before committing

## Important Notes

- The "Cliffhanger" section is the MOST important - it prevents losing mental context
- Be specific about error messages and file locations
- If session goals weren't discussed, infer from the conversation what was being worked on
- Always capture the "why" behind decisions, not just the "what"
- This report should make the next `/session-init` maximally useful
